# Protocol

Status: v0 orchestration protocol
Date: 2026-07-12

## Experiment Question

Can an obstruction/escape accounting structure classify divergent systems that collapse or endure through emergent structure, while preserving enough technical specificity to produce theorem targets, executable fixtures, or clean kill conditions?

## Primary Candidate North Star

> Develop a class-relative obstruction and escape accounting theory for open-ended systems, where emergence is tested by whether local admissibility survives attempted global/class closure through source growth, rule formation, boundary import, or productive residue, and collapse is classified by the way closure, completion, feedback, capture, overload, or source exhaustion absorbs that capacity.

## Experiment Design

### Step 1: Choose Divergent Examples

Select ten examples from different literatures:

- some formal;
- some computational;
- some biological;
- some institutional;
- some operational;
- some linguistic or cultural.

The examples must not all come from research repos. This tests whether the accounting object generalizes.

### Step 2: Fill Accounting Packets

For each example, fill:

- `ObstructionTransitionPacket`;
- `BoundaryImportPacket`;
- closure classification;
- completion-factorization test;
- no-go/escape/residue ledger.

### Step 3: Score Discrimination

Each example is scored from 0 to 3 on:

- field fillability;
- non-metaphor specificity;
- absorber resistance;
- fixture readiness;
- cross-domain usefulness;
- North Star relevance.

### Step 4: Identify Failure Modes

For each example, identify whether the accounting object is:

- strong;
- useful but derivative;
- under-specified;
- absorbed by an existing field;
- metaphorical;
- false fit.

### Step 5: Decide Next Move

Decision options:

1. Create repo now.
2. Run another bounded pre-repo experiment.
3. Keep as cross-repo interface artifact.
4. Reject as incoherent or duplicative.

## Pass Criteria

The experiment passes into "repo-preflight viable" only if:

- at least 7 of 10 examples can fill all major packet fields without forced analogy;
- at least 5 examples identify a concrete absorber or no-go class;
- at least 5 examples identify a nontrivial boundary import or residue;
- at least 3 examples can plausibly become theorem-oriented or executable fixtures;
- the closure typology prevents at least one false conclusion that "closure is collapse";
- the synthesis changes the North Star in a stricter way.

## Kill Criteria

The experiment should be killed or narrowed if:

- packetization is mostly descriptive and does not constrain claims;
- the same fields mean different things in every domain;
- the examples all reduce to known resilience, OEE, sheaf, distributed systems, or complexity theory;
- the strongest proposed invariant is representation-dependent;
- boundary import accounting merely says "everything depends on environment";
- no-go/escape ledgers do not compose or produce constraints.

## Orchestration Roles

These are roles for analysis, not separate repos or agents.

### Formal Skeptic

Looks for arbitrary class choices, hidden universes, modeler-added schemas, and non-invariant residues.

### Domain Translator

Checks whether the packet fields mean something real in each domain.

### Absorber Auditor

Asks which existing discipline already owns the example.

### Fixture Builder

Identifies whether a theorem, diagram, simulation, schema, or executable test could be built.

### North Star Editor

Updates the candidate North Star only when the examples force a stricter claim.

## Deliverables

This orchestration produces:

- a packetized example set;
- a scoring table;
- a list of strong and weak domains;
- a narrowed North Star;
- a next-run recommendation.
